# AegisFlow — AWS Infrastructure (Terraform)

This module provisions the complete AWS landing zone for AegisFlow:

- **VPC** with three-tier subnets across two AZs (public · private app · private DB)
- **ECS Fargate** cluster running the backend and frontend services
- **ECR** repositories with native image scanning enabled
- **RDS PostgreSQL 16** in private subnets, encrypted at rest
- **ALB + AWS WAFv2** with OWASP-aligned managed rule sets
- **AWS Secrets Manager** for DB credentials and JWT signing secret
- **S3** for SBOMs, scan reports, and incident attachments
- **CloudWatch** log groups and metric alarms
- **GuardDuty + Inspector v2** for continuous runtime threat detection
- **IAM OIDC provider** federating GitHub Actions (zero static AWS keys)

The Terraform code is itself scanned by **tfsec** as part of the CI pipeline.

## Prerequisites

1. **AWS account** with admin or PowerUser access for the initial bootstrap.
2. **Terraform** >= 1.6.
3. **AWS CLI** authenticated locally (`aws configure`) for the first apply.
4. **tfsec** for local IaC scanning (`brew install tfsec` / `choco install tfsec`).

## First-time bootstrap (local)

```bash
cd infrastructure/terraform

# 1. Configure your variables
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars to match your AWS account and preferences.

# 2. Initialize providers
make init

# 3. Plan and review
make plan

# 4. Apply (creates ~50 AWS resources, ~10 minutes)
make apply

# 5. Capture outputs for use in CI
make output
```

## Wiring to GitHub Actions

After `terraform apply`, copy the `github_actions_role_arn` output into your
repository secrets as **`AWS_ROLE_ARN`**:

```bash
gh secret set AWS_ROLE_ARN --body "$(terraform output -raw github_actions_role_arn)"
```

The `cd-deploy.yml` workflow uses this to assume the role via OIDC. No
long-lived AWS access keys are ever stored in GitHub.

You will also want to set:

```bash
gh secret set AWS_REGION --body "eu-west-3"
gh secret set ECR_BACKEND_URL  --body "$(terraform output -raw ecr_backend_url)"
gh secret set ECR_FRONTEND_URL --body "$(terraform output -raw ecr_frontend_url)"
gh secret set ECS_CLUSTER      --body "$(terraform output -raw ecs_cluster_name)"
gh secret set ECS_BACKEND_SVC  --body "$(terraform output -raw ecs_backend_service_name)"
gh secret set ECS_FRONTEND_SVC --body "$(terraform output -raw ecs_frontend_service_name)"
```

## Security posture

- **No public RDS** — DB subnets have zero internet route.
- **No static AWS keys** — GitHub Actions assumes a role via OIDC; trust policy
  restricts to our repo and allowed branches.
- **Secrets in Secrets Manager** — both DB password and JWT secret are
  generated by Terraform's `random_password`, never present in code or state
  in plaintext (state itself should live in encrypted S3 — see the commented
  backend block in `versions.tf`).
- **WAF in front of ALB** — three managed rule groups (Core, Bad Inputs, SQLi)
  + rate-limit rule.
- **ALB drops invalid headers** + HTTP→HTTPS redirect + TLS 1.3 policy.
- **ECR immutable tags** in production (overrideable per environment).
- **Container hardening** — non-root user (UID 1000), readonly filesystem for
  frontend, healthchecks defined.
- **GuardDuty + Inspector** for post-deploy continuous monitoring.

## Tearing down

```bash
make destroy
```

This will prompt for confirmation and remove all ~50 resources. Note that S3
buckets with content require manual emptying; the apply will fail if there are
attachments or scan reports — empty the bucket first via the AWS console or
`aws s3 rm s3://<bucket> --recursive`.
